# スケーリング方針（PoC → 全国展開）

> 関連 Issue: #6
> 対象: データ収集基盤を PoC（1自治体）から全国展開へ段階的に移行する計画

## 1. 背景

PoC は東京都のみ・GitHub Actions（cron）でデータ収集を行う（[decisions.md](./decisions.md) / [HANDOFF.md](./HANDOFF.md)）。全国展開では GitHub Actions 単体では限界があるため、段階的な移行方針を定める。

本ドキュメントは収集基盤（バックエンドのデータ取得）のスケーリングを扱う。配信レイヤー（CDN=jsDelivr→将来 R2、ホスティング=Cloudflare Pages、API=Supabase 直接）は各設計ドキュメントで確定済みで、本書はそれらと整合する。

## 2. GitHub Actions の制約の再整理（重要な補正あり）

HANDOFF.md は「実行時間の壁: 月 8,640 分 > 無料枠 2,000 分」を制約に挙げているが、**これは private リポジトリの場合のみ**である。

> **補正**: 本リポジトリは現在 **public** であり、**public リポジトリの GitHub Actions 実行時間は無制限・無料**。したがって「実行時間（分）の壁」は public である限り当面効かない。一方、以下の壁は **public / private を問わず効く**。

| 壁 | 内容 | public で効くか |
|---|---|---|
| 実行時間（分） | Free private は 2,000 分/月。5分ポーリングは 1 ワークフローでも月 8,640 分相当で超過 | **public では無制限（効かない）** |
| 同時実行数 | Free プランは同時実行ジョブ 20 まで | 効く |
| ジョブ最大実行時間 | 1 ジョブ最大 6 時間 | 効く |
| cron の信頼性 | スケジュール実行は高負荷時に遅延し得る／リポジトリが 60 日無操作だと無効化される | 効く |

**結論**: public のうちは「分」ではなく、**(a) cron の遅延・信頼性（時間に厳しい警報配信に影響）**、**(b) 同時実行 20 の上限**、**(c) 6 時間/ジョブ（重い SHP 変換）**、そして **(d) 将来 private 化した瞬間に分の壁が復活** が移行を駆動する要因となる。

## 3. 段階的移行とフェーズ別構成

| フェーズ | 規模 | 収集基盤 |
|---|---|---|
| フェーズ1（現在） | 1自治体（東京都） | GitHub Actions 単体 |
| フェーズ2 | 数自治体〜1県 | GitHub Actions + 処理分割（matrix / データ種別ごとの cron 分離） |
| フェーズ3 | 全国 | ハイブリッド: 高頻度・軽量 → Cloudflare Workers Cron / 低頻度・重い処理 → GitHub Actions（または コンテナ実行） |

### フェーズ3 の移行先方針

- **高頻度・軽量（警報 5 分ポーリング等）は Cloudflare Workers Cron Triggers に移行**する。ホスティング（Pages）・将来の CDN（R2）と同じ Cloudflare に揃え、エッジ基盤を一系統に保つ（[decisions.md](./decisions.md) のエコシステム統一方針と一致）
- **低頻度・重い処理（国土数値情報 SHP→GeoJSON 変換、GDAL 依存）は GitHub Actions のまま**とする。Workers は CPU 時間・実行時間・バイナリ依存（GDAL）に制約があり重いバッチ変換に不向きなため。必要に応じてコンテナ実行（Cloud Run jobs 等）も選択肢
- すなわちフェーズ3は「全部 Workers」ではなく、**更新頻度・処理特性でサービスを使い分けるハイブリッド**とする（HANDOFF.md の方針と一致）

> **コスト注記（重要）**: 全国規模（≒47 都道府県）の警報ポーリングを 1 つの Worker 実行で回す場合、Cloudflare Workers **無料プランの制約**（1 実行あたり サブリクエスト 50 / CPU 約 10ms）に当たりやすい。そのため、フェーズ3 の高頻度 Workers は実質 **Workers Paid（$5/月、サブリクエスト 1,000・CPU 最大 30 秒）が前提**になるか、**実行を分割**する必要がある。「Workers＝無料」を暗黙の前提にしないこと。

## 4. フェーズ別の移行トリガー条件

「何自治体になったら移行するか」を、規模だけでなく**制約に到達したか**で定義する。

### フェーズ1 → 2 の移行トリガー（いずれか）

- 対象が **複数都道府県**に拡大する
- 単一ポーリングジョブの 1 巡が cron 間隔に対して余裕を失う（例: 警報取得 1 巡が **90 秒**超 / 5 分間隔）
- ハザード SHP 変換ジョブが **1 ジョブ 6 時間**に接近する（目安: 連続 3 時間超）

### フェーズ2 → 3 の移行トリガー（いずれか）

- 並列ジョブ数が**同時実行上限 20 に恒常的に接近**する（目安: 定常的に 10 超）
- 5 分 cron の**遅延が頻発**し、警報の鮮度目標（検知から配信まで 5 分以内）を満たせなくなる
- **全国規模の高頻度ポーリングを開始**する（規模として全国 ≒ 47 都道府県）
- **リポジトリを private 化**する判断をする（= Actions 分が 2,000 分/月に制限され、5 分ポーリング単体でも超過するため、高頻度ポーリングは Actions から退避が必須になる）

## 5. フェーズ3 移行を見据えた PoC 実装上の制約事項

PoC 実装の時点で以下を守り、将来のクラウド移行コストを最小化する（HANDOFF.md の申し送りを具体化）。

1. **取得ロジックは自治体非依存の独立関数として切り出す**。エントリポイント（cron 実行単位）と取得ロジックを分離し、Workers へそのまま載せ替えられるようにする
2. **ランタイム移植性を保つ**。Web 標準 API（`fetch` / Web Crypto 等）を基本とし、Node.js 固有 API や OS 依存への依存を避ける（Cloudflare Workers ランタイム互換のため）。GDAL 等のネイティブ依存が必要な重い変換処理は最初から別エントリポイントに隔離する
3. **気象庁取得は都道府県ごとに別ジョブを立てず、1 ジョブ内のループで集約**する（同時実行数の壁を回避）
4. **更新頻度でデータ種別を分離**する。cron は警報=5分 / 避難場所=日次 / ハザード=週次〜月次 と頻度別に分け、頻度・処理特性に応じて将来の移行先を変えられるようにする
5. **差分検出（内容ハッシュ比較）で不要な書き込み・配信をスキップ**する（#9 / #5 で確立済み）。負荷とコストの両方を抑える
6. **状態をランナー外に置く**。GitHub Actions ランナーは ephemeral なので、最後に処理した `reportDatetime` 等の状態はローカルキャッシュではなく Supabase 等に保持する。これにより Workers 移行後も同じ状態管理が使える
7. **シークレットは環境変数経由で統一**する。GitHub Actions Secrets と Workers Secrets で同名の環境変数を用い、移行時にコード変更を不要にする（`.env.example` にキー名を集約）

## 7. 将来の申し送り

- フェーズ3 の高頻度収集は Cloudflare Workers Cron に移行し、重いバッチ変換は GitHub Actions / コンテナに残すハイブリッド構成とする
- フェーズ3 の高頻度 Workers は無料枠の制約（サブリクエスト 50 / CPU 約 10ms）に当たりやすいため、Workers Paid（$5/月）への昇格または実行分割をコストに織り込むこと
- 本書は収集基盤を対象とするが、全国規模では **Supabase 側のスケール（DB 容量・コネクション数・無料枠上限）も頭打ちになる**ため、Supabase の有料プラン昇格を並行して検討すること
- private 化を検討する場合は、その前に高頻度ポーリングを Actions から Workers へ退避させること（分の壁が即座に効くため）
- 移行トリガーの閾値（90 秒 / 6 時間接近 / 同時 10 超 等）は実測に基づき調整する
