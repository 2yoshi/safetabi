# CDN・静的 GeoJSON 配信設計

> 関連 Issue: #5
> 対象: 低頻度更新のハザードデータ（洪水・土砂・津波ポリゴン、避難場所一覧）の静的配信方式

## 1. 背景と目的

SafeTabi のデータは更新頻度によって最適な配信レイヤーが異なる。

| 区分 | データ | 更新頻度 | 配信レイヤー |
|---|---|---|---|
| 高頻度・空間検索あり | 警報・避難指示、避難場所の近傍検索 | 5分〜日次 | Supabase（PostgreSQL + PostGIS）REST API |
| **低頻度・面データ** | **洪水・土砂・津波ハザードポリゴン** | **週次〜月次** | **静的 GeoJSON + CDN（本ドキュメントの対象）** |

ハザードポリゴンは「更新は稀だが、サイズが大きく、地図描画でそのまま使う面データ」である。これを PostGIS に載せて毎回 API で取得するのは帯域・コストの無駄が大きい。**ビルド成果物として静的ファイル化し、CDN から配信する**のが適切。

## 2. 要件

- **コスト**: PoC は無料枠内で運用する（[decisions.md](./decisions.md) のコスト最小化方針）
- **オフライン対応**: PWA Service Worker からキャッシュ取得できること（[HANDOFF.md](./HANDOFF.md) のオフライン必須要件）
- **CORS / HTTPS**: ブラウザ・アプリから直接 fetch できること
- **更新検知**: 週次〜月次の更新をクライアントが検知し、キャッシュを更新できること
- **GitHub Actions 統合**: 既存のデータ収集パイプライン（cron）から成果物を配置できること

## 3. 配信方式の選定

### 候補比較

| 方式 | コスト | GitHub Actions 統合 | 容量・ファイルサイズ制限 | CDN/CORS | 備考 |
|---|---|---|---|---|---|
| **GitHub Pages + jsDelivr** | 無料 | ◎（同一リポジトリにコミットするだけ） | jsDelivr: 1ファイル 50MB まで | ◎ 標準対応 | PoC に最適 |
| Cloudflare R2 + CDN | 無料枠（10GB保存/月） | ○（API/CLI でアップロード） | 実質無制限 | ◎ | 大容量・全国展開向き |
| Supabase Storage | 無料枠（1GB） | ○ | 無料枠が小さい | ○ | 高頻度データと配信特性が異なり混在は非推奨 |

### 決定: **GitHub Pages + jsDelivr（PoC フェーズ）**

理由:

1. **完全無料**で、既存の GitHub Actions パイプラインからコミットするだけで配置できる（追加の認証情報・SDK 不要）
2. jsDelivr が GitHub リポジトリを自動でグローバル CDN 配信し、**HTTPS・CORS・gzip/brotli 圧縮**を標準で提供する
3. 東京都のみの PoC スコープでは、後述の分割により 50MB/ファイル 制限に十分収まる
4. 後から R2 への移行は「アップロード先の変更」に閉じ、クライアントの取得ロジックは URL ベースのまま維持できる

### 配置とブランチ運用

- 成果物は専用ブランチ `data` （または `gh-pages`）に配置し、アプリのソースコードと混在させない
- jsDelivr の参照 URL 形式:
  ```
  https://cdn.jsdelivr.net/gh/<owner>/safetabi@<commit-or-tag>/<path>
  ```
- **キャッシュ無効化のためタグ／コミットハッシュで参照する**（`@latest` やブランチ名はキャッシュ TTL が長く更新が即時反映されないため、本番取得には使わない）

## 4. ファイル分割・タイル化の判断

### ディレクトリ・命名規約

災害種別 × 行政区画でファイルを分割する。

```
hazard/
├── manifest.json                       # 全ファイルの索引（バージョン・ハッシュ・パス）
├── flood/                              # 洪水浸水想定区域
│   └── 13/                            # 都道府県コード（東京都 = 13）
│       ├── 13101.geojson              # 市区町村コード（千代田区）
│       ├── 13102.geojson
│       └── ...
├── landslide/                         # 土砂災害警戒区域
│   └── 13/...
├── tsunami/                           # 津波浸水想定区域
│   └── 13/...
└── shelter/                           # 指定緊急避難場所（日次・小サイズ）
    └── 13.geojson
```

### 分割粒度: **市区町村単位**

理由:

1. **jsDelivr の 50MB/ファイル 制限**を回避する（都道府県全域の洪水ポリゴンは数十MBに達しうる）
2. **必要分のみ取得**できる。利用者は現在地周辺の市区町村だけ取得すればよく、通信量とオフラインキャッシュ容量を抑えられる
3. **更新の局所化**。一部市区町村のみ更新された場合、差分のあるファイルだけ再配信され、未変更ファイルの CDN キャッシュが維持される

避難場所一覧（小サイズ・日次）は都道府県単位の 1 ファイルで十分。

### タイル化（PMTiles / MBTiles）: **PoC では不要**

- PoC（東京都・市区町村分割）の想定ファイルサイズは PMTiles 導入コストに見合わない
- 以下の **移行閾値**のいずれかに達した時点で PMTiles 化を再検討する:
  - 単一 GeoJSON ファイルが **10MB** を超える（地図描画時のパース・描画が重くなる目安）
  - 全国展開フェーズに入り、ファイル総数・総容量が CDN 配信に不向きになる
  - ズームレベルに応じたポリゴン簡略化（LOD）が必要になる

## 5. 更新検知・キャッシュ戦略

### manifest.json

CDN ルートに索引ファイルを置き、クライアントはこれを起点に各ファイルを取得する。

```json
{
  "version": "2026-06-21T00:00:00Z",
  "files": [
    {
      "path": "hazard/flood/13/13101.geojson",
      "hash": "sha256:...",
      "bytes": 1234567,
      "updated_at": "2026-06-21T00:00:00Z"
    }
  ]
}
```

- **manifest.json は短い TTL（またはコミット固定 + 定期ポーリング）**で取得し、`hash` の差分で更新を検知する
- データ本体は **バージョン（コミットハッシュ／タグ）付き URL** で取得し、`Cache-Control: immutable` 相当の長期キャッシュとして扱う
- データ収集パイプラインは内容ハッシュを比較し、**変更がなければ再コミットしない**（[HANDOFF.md](./HANDOFF.md) の差分検出方針と一致）。これにより未変更ファイルの CDN キャッシュとクライアントキャッシュが維持される

### PWA Service Worker

- 初回アクセス時に現在地周辺の市区町村ファイルをプリキャッシュ
- manifest の `hash` 変化を検知したファイルのみ再取得（stale-while-revalidate）
- オフライン時はキャッシュ済みファイルで地図描画を継続

## 6. 決定事項（Issue #5 のチェックリスト）

- [x] **配信方式を選定する** → GitHub Pages + jsDelivr（PoC）。将来は Cloudflare R2 へ移行可能な設計
- [x] **ファイル分割・タイル化の要否を判断する** → 災害種別 × 市区町村単位で分割。タイル化（PMTiles）は PoC 不要、移行閾値（単一ファイル 10MB 超 / 全国展開）で再検討

## 7. 将来の申し送り

- 全国展開時は jsDelivr の 50MB 制限・ファイル総数の観点から **Cloudflare R2 + CDN** へ移行する。配置先 URL を差し替えるだけで済むよう、クライアントの取得は manifest.json 起点の相対パス解決にしておく
- ズーム対応の簡略化が必要になったら **PMTiles** を導入し、`hazard/*.pmtiles` を併設する（GeoJSON とは別系統で配信）
