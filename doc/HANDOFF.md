# SafeTabi プロジェクト引き継ぎドキュメント

> このドキュメントは Claude (claude.ai) との企画検討セッションから
> Claude Code への引き継ぎ用にまとめたものです。

## プロジェクト概要

**プロジェクト名**: SafeTabi
**事業内容**: 訪日外国人観光客向けの災害対応・避難支援サービス
**位置づけ**: 日本政府・自治体オープンデータを活用したビジネスベンチャー
**成功の形**: 実用的なプロダクト（スマートフォンアプリ＋Webアプリ）のリリース

### ターゲットユーザー
- 訪日外国人（インバウンド）
- 課題: 言語バリア、地理的不慣れ

### マネタイズモデル（検討中）
- BtoG（自治体向け）
- BtoB（観光事業者向け）

---

## サービスコンセプト（優先順位順）

1. **リアルタイム災害アラート表示**
2. **最寄り避難所・避難ルート案内**
3. **現在地ハザードリスク情報**
4. **多言語対応**（英語・簡体字中国語・繁体字中国語・韓国語）

### プラットフォーム
- スマートフォンアプリ（iOS/Android）
- Webアプリ（ブラウザ）
- 両方を提供予定

## 技術アーキテクチャ

### データソース（すべて無償・商用利用可）

| データ | 提供元 | 形式 | 更新頻度 |
|---|---|---|---|
| 警報・避難指示（警戒レベル1〜5） | 気象庁 JSON API | JSON | リアルタイム（5分毎想定） |
| 緊急地震速報 | 気象庁 | JSON | リアルタイム |
| 指定緊急避難場所一覧 | 国土地理院（GSI） | CSV（緯度経度・災害種別フラグ付き） | 日次想定 |
| 洪水浸水想定区域 | 国土数値情報 | Shapefile | 週次〜月次想定 |
| 土砂災害警戒区域 | 国土数値情報 | Shapefile | 週次〜月次想定 |
| 津波浸水想定区域 | 国土数値情報 | Shapefile | 週次〜月次想定 |

**気象庁JSON APIのURL構造例**:
```
https://www.jma.go.jp/bosai/warning/data/warning/{area_code}.json
https://www.jma.go.jp/bosai/forecast/data/forecast/{area_code}.json
```
area_code はエリアコード（例: 東京 = 130000）。政府標準利用規約に準拠して利用可能。

### 機能レイヤー構成

```
ユーザー（スマホ / Web）
  ↓
① リアルタイム災害情報（警報・避難指示・地震速報）
  ↓
② 最寄り避難場所への経路（GPS取得 → 災害種別マッチング → 経路案内）
  ↓
③ 現在地のハザード情報（洪水・土砂・津波ポリゴンとGPSの空間演算）
  ↓
④ 多言語対応（UI翻訳・アラート翻訳・ピクトグラム表示）
```

### 重要な技術要件
- **オフライン対応が必須**: 観光地・山岳エリアは通信環境が不安定なため、避難所データとハザードポリゴンはPWA（Service Worker）でローカルキャッシュする設計が必要
- **プッシュ通知**: FCM（Android）/ APNs（iOS）経由、気象庁API更新をトリガーに配信
- **多言語化**: LLMによるリアルタイム翻訳 + ピクトグラム案内（言語に依存しない視覚案内）

## データ自動収集アーキテクチャ

### PoCスコープ
- **規模**: 特定1自治体のみ（小規模検証）
- **重視点**: コスト最小化（無料枠内で運用）
- **スケジューリング基盤**: GitHub Actions（cron）

### PoC構成（フェーズ1）

```
GitHub Actions（cron）
  ↓
収集スクリプト（Python）: 気象庁JSON / GSI CSV / 国土数値情報SHP
  ↓
正規化・変換（統一GeoJSONスキーマ、SHP→GeoJSONはGDAL/ogr2ogr使用）
  ↓
差分検出（ハッシュ比較、変更なければ書き込みスキップ）
  ↓
ストレージ
  - Supabase（PostgreSQL + PostGIS）: 高頻度更新データ、空間検索が必要なもの
  - 静的GeoJSON + GitHub Pages/CDN: 低頻度更新データ（ハザードポリゴン等）
  ↓
配信レイヤー
  - REST API（Supabase自動生成）
  - CDN経由配信（GitHub Pages / jsdelivr）
  ↓
アプリ（スマホ / Web）: リアルタイム表示 + オフラインキャッシュ
```

### データ取得頻度設計

| データ | 取得間隔 |
|---|---|
| 気象庁 警報・避難情報 | 5分毎 |
| 国土地理院 避難場所CSV | 日次 |
| 国土数値情報 ハザードSHP | 週次〜月次 |

### 全国展開時の制約と対応方針（重要・将来設計への申し送り）

**GitHub Actions単体では全国規模に対応できない**ことが判明している。理由:

1. **実行時間の壁**: 気象庁5分毎ポーリング × 全国スコープ → 月8,640分超（プライベートリポジトリ無料枠2,000分を大幅超過）
2. **同時実行数の壁**: Freeプランの同時実行ジョブ数は20まで。都道府県別に並列化するとすぐ到達
3. **ジョブ最大実行時間の壁**: 1ジョブ最大6時間。全国データを1ジョブで処理すると重い変換処理（SHP→GeoJSON）で引っかかる可能性

**推奨する段階的移行方針**:

| フェーズ | 規模 | 基盤 |
|---|---|---|
| フェーズ1（現在のPoC） | 1自治体 | GitHub Actions単体 |
| フェーズ2 | 数自治体〜1県 | GitHub Actions + 処理分割 |
| フェーズ3 | 全国 | GitHub Actionsは「CI/CDトリガー・軽量タスク」に役割変更。高頻度・大量並列の収集処理はCloud Run jobs / Cloud Functions（GCP）または Cloudflare Workers Cron Triggers に移行 |

**実装上の申し送り**: フェーズ3移行を見据えて、**今のPoC実装時点から「気象庁取得ロジック」を自治体に依存しない独立した関数として切り出しておくこと**。これにより将来のクラウド移行コストを最小化できる。

回避策の具体例:
- 気象庁データは都道府県ごとに別ジョブを立てず、1ジョブ内でループ処理して全国を1リクエスト相当に集約する
- 更新頻度に応じてサービスを使い分ける（高頻度軽量 → Cloud Functions等、低頻度重い処理 → GitHub Actionsのまま）

## リポジトリ構成案

プロジェクト名は **SafeTabi** で確定。

| リポジトリ名 | 役割 |
|---|---|
| `safetabi` または `safetabi-app` | アプリ本体（スマホ＋Web） |
| `safetabi-data-collector` | オープンデータ自動収集（GitHub Actions） |
| `safetabi-api` | 配信API（Supabase連携。将来分離する場合） |

**PoCフェーズの開始方針**: `safetabi-data-collector` 単体、または `safetabi` モノレポ（app/data-collector をディレクトリ分割）のどちらかから開始するのが現実的、という結論で議論が止まっている。Claude Codeでの作業開始時にどちらにするか最終決定が必要。
